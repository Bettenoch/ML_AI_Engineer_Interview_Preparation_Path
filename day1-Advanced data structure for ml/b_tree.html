🔷 Node
A single unit in the B-tree that contains keys and references to child nodes. Each node has a maximum capacity defined by the tree's order.

Example: [20, 30] is a node with 2 keys.
🌳 Tree
The entire B-tree structure consisting of all nodes organized in a hierarchical manner. It maintains balance and allows efficient searching, insertion, and deletion.
👶 Children
Nodes that are directly connected below another node. In a B-tree of order m, each internal node can have at most m children.

Rule: Node with k keys has k+1 children.
🍃 Leaf Node
Nodes at the bottom level of the tree that have no children. All leaf nodes are at the same level in a B-tree, ensuring balance.

Shown in green in the diagram above.
🌟 Root
The topmost node of the tree. It's the entry point for all operations. The root can have fewer keys than other nodes (minimum 1 key).

Shown in red in the diagram above.
📦 Non-Full Child
A node that has fewer than the maximum allowed keys. These nodes can accommodate more keys without splitting.

Example: [25] has only 1 key when max is 2.
⚠️ Full/Overflowing Child
A node that has reached its maximum capacity of keys. Attempting to add more keys requires splitting the node.

Shown in yellow with pulsing animation above.
✂️ Splitting Child Nodes
When a full node needs to accommodate a new key, it splits into two nodes. The middle key moves up to the parent node.

Process: Full node → Split → Two half-full nodes + middle key promoted.
🔢 Maximum Keys
The maximum number of keys a node can hold, determined by the tree's order. For a B-tree of order m, max keys = m-1.

In our example: Order 3 → Max 2 keys per node.
🔗 Maximum Children
The maximum number of child nodes an internal node can have, equal to the tree's order m.

Formula: Max children = Order of B-tree
In our example: Order 3 → Max 3 children per node.
